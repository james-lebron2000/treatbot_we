## 2. 三方辩论收敛的关键决策（前置阅读）

下面 5 项决策影响多个任务，单独提炼，避免在每个任务里重复。

### 2.1 CRO–trial–site 归属采用 site 维度，**不引入 share_percent**

- 同 NCT 在不同 site（研究中心）由不同 CRO 运营是 GCP 真实场景，**不是 bug 是建模缺失**
- 同 site 强制 100% 归属一家 CRO，避免分账复杂度
- 用户提交申请时**只选 trial、不选 site**（避免漏斗损失 30-40%）
- 后端按"用户城市 → 最近活跃 site → 该 site 当前 CRO"三步路由
- 500km 内无 site → 进 `pending_geo_review` 状态，不直接拒绝（医学硬要求：晚期患者宝贵窗口期）
- `icf_signed` 时 `site_id_confirmed` 必须非空，否则状态机硬阻断

### 2.2 OCR 双阈值 v0 + biomarker 12 项强制人工审核白名单

- 阈值 v0：`< 0.75` 丢弃 / `≥ 0.95` 直通 / `0.75-0.95` needs_review
- 4 周后用真实 confidence 直方图回灌调整
- **biomarker 白名单（12 项）无视 confidence，100% 进人工队列**：EGFR / ALK / ROS1 / KRAS G12C / HER2 / PD-L1 / MSI-H / BRCA / TNM 分期 / 病理类型极性 / PD-1 既往用药 / 妊娠状态
- AI 初筛结果秒级展示（保留 C 端体验），但**点击"申请"前必须 gate 在人工复核完成**（医学硬线，违反即患者错配事件）
- 人工审核 SLA：工作日 12h / 周末 24h / 最长 36h
- 审核团队：日均 1000 份估算需 8 人，冷启动期日均 ≤200 份用 2 人外包

### 2.3 状态机：ICF 是 action 事件 + status 守卫的混合实现

- 不把 ICF 拆成独立 status（避免状态图膨胀）
- 新增 `screening` status 覆盖 contacted → enrolled 之间的窗口
- 最终 status 链：`pending → contacted → screening → enrolled / screen_failed / withdrawn`
- `enrolled` 写入时**硬校验**同 application 下存在 `icf_signed` 且 timestamp 早于 enrolled，否则拒绝（GCP 强制）
- `action_event` 17 项枚举（5 组：触达 / 知情同意 / 筛选 / 入组与给药 / 撤回）
- 三层防御：应用层 stateMachine.canEmit + DB CHECK 约束 + 影子审计

### 2.4 非肿瘤词典首批 3 个方向：T2DM / 慢乙肝 / RA

- 选取依据：在华活跃试验数 + CRO 已签约意向 + 词典构建成本
- 第二批 Q4 末启动：心衰 + 神经退行（依赖 OCR 影像报告解析能力扩充）
- **每个方向 day 1 必须有禁忌症字段**，缺一不可上线（医学硬线）
- 词典 v1 合格量化指标：召回率 ≥0.90 / 准确率 ≥0.92 / 禁忌症字段召回率 ≥0.98 / 禁忌症 FN < 0.02
- 词典扩展不在 Q4 P0 范围（P1 启动框架，P2 数据飞轮），但相关基础设施（命中率监控）在 Q4-T0-10 一并交付

### 2.5 测试基础设施先行

- `tests/api.test.js` 改为 CI 必跑（fast + slow 两条流水线）
- 集成测试 flakiness 控制：每 PR 独立 schema、`jest.useFakeTimers()`、nock 强制
- schema diff CI job（`migra` / `squawk`）：staging vs prod 每日跑
- metric 心跳自检：每分钟 heartbeat counter，absent >2min P1 告警
- 任何 P0 修复合并必须先于本任务，否则没有回归保护

## 3. P0 任务详细 PRD

### Q4-T0-1 · trialCrawler null 覆盖防护

**修改项**
- [server/jobs/trialCrawler.js](../server/jobs/trialCrawler.js) `diffAndApply()` —— 增加 null 守门
- [server/services/clinicalTrialsClient.js](../server/services/clinicalTrialsClient.js) —— 区分"上游显式 null"vs"上游字段缺失"
- 新增 `trial_field_change_review` 表 —— 待人工复核队列
- 新增 metric `crawler_field_null_total{field}`

**目的**
Q3 上线的 trialCrawler 上游字段 null 会**静默覆盖**库内已有的真值。ClinicalTrials.gov 偶发字段缺失（`enrollment.count`、`phases` 在某些状态下就是空的），当前会把库里的 `enrolled_count=120` 刷成 NULL。后果：matchEngine 拿到 NULL 招募人数，可能继续向晚期患者推荐已实际关闭的试验，**患者放弃当下治疗去等待一个不存在的机会，是延误治疗的实质性伤害**（医学专家定为 Grade 4）。

**产品视角（PM）**
- 数据资产慢性污染：3 个月后回看，匹配模型训练样本一半不能用
- "假承诺"导致小红书差评：用户花 5 分钟传病历 → 申请 → 才发现试验早关闭
- 修复后才能给后续 C10 匹配 trace、C11 缓存失效提供干净底座

**用户视角（患者 / 家属）**
- 不再看到"已招满"的试验出现在推荐列表
- 招募人数显示真实剩余，不会因为字段被刷空而看到"未知"

**CRO / 客户视角**
- 不再被推送"招募人数错乱"的脏申请
- 试验状态变更走人工复核队列，CRO 端可看到"上游某字段疑似异常"提示，对账时有据可查

**完成后获益（量化）**
- ✅ `crawler_field_null_total` 单字段 null 率 < 5%
- ✅ trial_field_change_review 表每日待审 < 30 条（人工 1h 内可处理完）
- ✅ 用户在"已关闭试验"上花费的中位时长（提交申请到收到拒绝）从 ~3 天降到 0（不再展示）

### Q4-T0-2 · CRO–trial–site 归属重建

**修改项**
- 新建表 `cro_trial_ownership(id, cro_id, trial_id, site_id, effective_from, effective_to, version, created_at)` + UNIQUE(site_id, effective_from)
- 新建表 `trial_sites(id, trial_id, site_name, city, address, lat, lng, is_active)`
- 移除 `cro_companies.trial_ids` JSON 列（W9 周）
- 新增 `services/croOwnershipResolver.js`：`resolveCroForApplication(userCity, trialId, submittedAt)` 确定性函数
- [server/services/applicationStateMachine.js:180](../server/services/applicationStateMachine.js) `findCroCompanyByTrialId` 替换为 `findOwnersByTrialId(trial_id, application_site_id)`
- [server/services/billing.js](../server/services/billing.js) computeMonthly 按 site 维度归属
- 新增 status `pending_geo_review`（500km 内无 site 时进入）
- 新增 metric `application_owner_assign_total{strategy=auto|manual}`
- 新增 metric `cro_ownership_shadow_diff_total`（迁移期影子读 diff）
- 9 周灰度迁移：W1-2 双写 → W3 影子读 → W4 10% → W5 50% → W6 100% → W7-8 JSON 只读 → W9 删字段

**目的**
Q3 上线的 CRO 归属基于 `cro_companies.trial_ids JSON` 数组，命中多家时**按 MySQL 物理行序随机归一**。这意味着：
- 计费金额取决于行序，非确定性
- 一旦同一 trial 被两家 CRO 认领（多中心试验里很常见），等于平台"卖了同一份线索两次但只收一次钱、还随机给一家"——商务事故
- 现实里同一 NCT 在不同 site 由不同 CRO 运营是 GCP 常态，是建模缺失不是 bug

**产品视角（PM）**
- 商务可解释性：CRO 能从合同回看"我负责北京中心、上海归 CRO-B"，一目了然
- 不引入 share_percent：合同复杂度、退单追溯都简化
- 用户不感知 site 选择：漏斗损失保护

**用户视角（患者 / 家属）**
- 提交申请时只选 trial（不选 site），保持原交互
- 系统自动按用户城市推荐最近 site
- 500km 内无 site → 不直接拒绝，进 `pending_geo_review`，CRO 决定是否承担差旅（II 期以上肿瘤试验通常承担）

**CRO / 客户视角**
- 同 NCT 不同 site 不同 CRO 不冲突，符合多中心试验真实结构
- 按 site 计费清晰，对账有据可查
- ICF 签署时 `site_id_confirmed` 必须非空 → CRO 必须在 contacted → screening 之间显式分配 site，避免 GCP "归属不清=责任不清"问题

**完成后获益（量化）**
- ✅ `application_owner_assign_total` sticky 命中率 > 99%
- ✅ 跨月计费 100% 可重算（同输入恒同输出）
- ✅ `pending_geo_review` 状态处理 SLA：CRO 48h 内决策
- ✅ 9 周灰度无业务中断

### Q4-T0-3 · CPA 计费版本快照

**修改项**
- 新建表 `cro_pricing_history(cro_id, cpa_qualified_status, cpa_price, effective_from, effective_to, changed_by, changed_at, change_reason)`
- 新建表 `billing_snapshot(year_month, cro_id, trial_id, site_id, qualified_count, unit_price, amount, frozen_at)`
- `cro_companies.cpa_*` 字段保留作"当前生效"，UPDATE 时同步 INSERT history 行
- [server/services/billing.js](../server/services/billing.js) `computeMonthly()` 按事件 `created_at` 反查 history（`effective_from <= event_time < effective_to`）
- 月初 cron：每月 1 号 02:00 北京时间冻结 `billing_snapshot`，对账以 snapshot 为准
- 运营后台 CPA 阈值变更必须二次确认弹窗，文案明确"仅对未来线索生效"
- 新增 metric `cpa_pricing_version_in_use`（gauge）+ `cpa_recompute_drift_amount`（histogram）

**目的**
Q3 上线的 CPA 计费用 `cro_companies.cpa_qualified_status` 当前值做匹配。如果运营 5/15 把 `screened` 改成 `enrolled`，回头算 5 月账单，**5/1–5/14 的 screened 事件会全部漏算**（或反过来多算）——直接动收入命脉。

**产品视角（PM）**
- 收入正确性是商业化第一信任：CRO 拒付一次，平台收入确认逻辑会被法务要求重做，比从一开始就锁版本贵 10 倍
- 阈值变更需二次确认弹窗，避免运营误操作

**用户视角（患者 / 家属）**
- 用户不感知

**CRO / 客户视角**
- 月度账单可对账：CRO 自己手算的金额和平台一致
- 阈值变更只影响未来线索，历史账单不被回溯改写
- 月初冻结 snapshot 给财务对账提供"权威版本"

**完成后获益（量化）**
- ✅ `cpa_recompute_drift_amount` p99 < 0.01 元
- ✅ 跨月场景 100% 可重算（5/15 改阈值，5 月账单不变）
- ✅ `billing_snapshot` 与实时 `computeMonthly` 跨月对比金额漂移 = 0
- ✅ 第一笔 CRO 月度账单复核 100% 通过

### Q4-T0-4 · 状态机 transition / touchpoint 拆分 + ICF 节点

**修改项**
- 新增表 `application_action_event(id, application_id, action, actor_role, actor_id, occurred_at, payload, source_doc_id, schema_version)`
- 新增 status：`screening`（contacted 与 enrolled 之间）
- 状态链：`pending → contacted → screening → enrolled / screen_failed / withdrawn`
- 移除 `cancelled` 状态（Q4-T0-4 配合 P1 完成数据回填）
- [server/services/applicationStateMachine.js](../server/services/applicationStateMachine.js) 新增 `recordAction(applicationId, action, actor, payload)`
- 新增 `services/actionStateMachine.js`：`canEmit(currentActions, newAction)` 应用层校验
- DB 层 trigger：`enrolled` INSERT 前校验存在 `icf_signed` 且 timestamp 早于 enrolled
- `transition()` 在所有 controller 调用点同时调 `recordAction`，无论是否真流转
- ajv JSON Schema 契约校验在 controller 入口 + DB 写入前各跑一次
- 新增 metric `application_touchpoint_total{action}` + `icf_audit_violation_total`

**action 枚举（17 项，5 组）**

| 组 | action | 必填 payload |
|---|---|---|
| 触达 | `contact_attempted` | channel, time |
| 触达 | `contact_succeeded` | - |
| 触达 | `contact_failed_3x` | last_attempt_at（自动转 dead_lead） |
| 知情同意 | `icf_version_presented` | icf_version |
| 知情同意 | `icf_signed` | signed_at, signed_location, icf_version, is_remote_e_sign |
| 知情同意 | `icf_declined` | reason_code |
| 筛选 | `screening_visit_scheduled` | scheduled_at, site_id, planned |
| 筛选 | `screening_visit_completed` | completed_at, site_id |
| 筛选 | `screening_passed` | - |
| 筛选 | `screening_failed` | reason_code (ICH-E3 标准) |
| 入组 | `enrolled` | randomization_id |
| 入组 | `first_dose_administered` | dose_at, dose_amount |
| 不良事件 | `ae_reported` | ctcae_grade, sae_flag |
| 不良事件 | `sae_reported` | reported_to_irb_within_24h |
| 撤回 | `withdrawn_by_subject` | reason_code |
| 撤回 | `withdrawn_by_site` | reason_code (AE/无效/方案违背) |
| 备注 | `note_added` | note_text |

**目的**
Q3 状态机里 `from === to` 直接短路返回，不留事件——CRO 实际业务里"今天联系一次、3 天后再联系一次"两次都点 contacted 现在第二次没留痕。后果：
- GCP audit trail 要求 PI 对 site 入组的每位受试者负责，"重复 contacted 不留痕"意味着无法证明 CRO 联系患者的真实次数，监管核查会判 critical finding
- CRO 触达 KPI 算不出"真实触达次数"，会自己另记 Excel，平台从"工作台"退化成"提交器"，DAU 立刻塌
- ICF 签署没显式节点，违反 ICH-E6(R2) 4.8.8 与中国 GCP 2020 版第 23 条"知情同意先行"原则

**产品视角（PM）**
- 同状态二次操作允许，但归类为 `touchpoint` 而非 `transition`，时间线上以次级样式展示，不影响 SLA 计时
- 用户端 4 里程碑文案规避医学伤害措辞（见 Q4-T0-9）

**用户视角（患者 / 家属）**
- 不感知底层 17 个 action，但能感知"研究团队已多次尝试联系您"等友好文案
- ICF 在用户端不单独占里程碑（避免法律文书吓退用户），但在"研究团队已联系"详情页内嵌"待签署知情同意书"任务卡

**CRO / 客户视角**
- 时间线展示完整 17 个 action，绩效统计有据
- ICF 阻断逻辑保护合规：不会出现"还没签同意书就被标记 enrolled"的乌龙
- SAE 24h 强制上报标记，监管核查时一眼看到合规

**完成后获益（量化）**
- ✅ `application_touchpoint_total{action}` 完整覆盖 17 项
- ✅ `icf_audit_violation_total` = 0
- ✅ enrolled 100% 有前置 icf_signed
- ✅ CRO 触达 KPI 与 action_event 表对齐 100%

### Q4-T0-5 · 申请去重（30 天 + 并发安全）

**修改项**

### Q4-T0-11 · 集成 CI 必跑 + 心跳自检 + schema diff

**修改项**
- `tests/api.test.js` 改为 CI 必跑（service-container MySQL + Redis）
- CI 流水线分两条：
  - **fast**（lint + 296 unit tests, ~3s, blocking）—— PR 通过即允许 merge
  - **slow**（api.test.js + 集成 ~3min, required-but-non-blocking）—— 失败自动开 issue + Slack on-call + 30min 内不修复触发 revert bot
- nightly 跑 full e2e
- 新增 `metrics_self_check`：每分钟自增一个 heartbeat counter
- Alertmanager 规则：heartbeat absent >2min → P1 告警
- 新增 schema diff CI job：
  - `pg_dump --schema-only $PROD_REPLICA > prod.sql`
  - `psql -f migrations/*.sql $TEST_DB && pg_dump --schema-only $TEST_DB > test.sql`
  - `migra prod.sql test.sql --unsafe || (echo "schema drift" && exit 1)`
  - 工具：`migra` + `squawk`（lint 危险迁移如无 IF NOT EXISTS、加锁列）
- Flakiness 控制：
  - 每 PR 独立 schema：`treatbot_test_${PR_NUM}_${SHA}`，测试结束 drop
  - fixture 用 `truncate ... restart identity cascade`
  - jest `--testRetries=2`，连续 flaky 3 次自动加 `@flaky` 标签 + 发 issue
  - 时间相关用例必须 `jest.useFakeTimers()`
  - 网络 mock 强制 nock，未 mock 请求直接 fail

**目的**
本季度 P0 修复 11 项中很多涉及核心流程改动，**没有真集成回归网，"修一个崩三个"是必然**。Q3 上线时 296 tests 全绿但 `tests/api.test.js` 默认排除 = 实质是放大的单测，prom-client 静默归零无人发现，schema 漂移无人监控——任何 P0 修复都建立在不可靠的回归网上。

**产品视角（PM）**
- 集成 CI 上线后，所有 P0 任务才有"灰度回滚可逆"保障
- 心跳自检防止 CPA 计数器静默归零

**用户视角（患者 / 家属）**
- 用户不感知，但平台稳定性提升直接影响体验

**CRO / 客户视角**
- 平台 SLA 提升 → CPA 计费、状态流转可信度提升
- schema diff 防止 staging vs prod 漂移导致的"在 staging 测过的功能到 prod 失效"事故

**完成后获益（量化）**
- ✅ 任何 PR 必须 fast + slow 两条都绿才能合
