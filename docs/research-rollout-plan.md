# Treatbot 下一步研究落地方案（2026-04-15）

## 1. 当前基线

- 当前本地 `main` 已与 `origin/main` 对齐，基线提交为 `aadb35c`。
- 患者端已经具备完整主链路：上传图片/PDF/聊天文件、结构化病历展示、解释式匹配、报名提交。
- 匹配结果页已经支持 `入组命中 + 排除风险 + 缺失证据` 三栏解释，适合进入真实预筛场景。
- 文档与代码都表明当前系统更接近“可演示的预筛平台”，距离“可稳定运营的研究招募工具”还差三类能力：
  - 数据质量闭环：缺少金标准样本、人工复核、结果回流。
  - 运营闭环：报名后缺状态流、缺专人分配、缺联系结果沉淀。
  - 评估闭环：缺量化指标，无法判断 OCR、结构化、匹配是否真的有效。

## 2. 下一阶段总目标

目标不是继续铺功能，而是把产品收敛成一个可验证的研究招募 MVP。

建议未来 6-8 周只做一件事：把 Treatbot 从“患者上传后给出候选试验”推进到“患者上传后，系统完成初筛，研究团队可接手跟进，并能衡量筛选质量”。

默认试点病种收敛为 3 类：

1. 非小细胞肺癌
2. 乳腺癌
3. 结直肠癌

原因很直接：试验量足够、入排标准相对标准化、结构化字段复用率高，适合先做验证。

## 3. 这轮落地的核心假设

1. 患者端不直接承诺“给出最终结论”，而是提供可信的初筛和资料整理。
2. 真正的研究落地要靠“AI 初筛 + 人工复核 + 研究协调员联系”三段式，而不是纯算法闭环。
3. 先提升 Top-N 候选质量，比追求全癌种覆盖更重要。
4. 先跑通一个可测量的试点闭环，再扩展到更多病种和更多中心。

## 4. 分阶段方案

## Phase 1：建立研究级数据底座（第 1-2 周）

目标：让系统知道自己“判得准不准”。

交付物：

1. 建立 100-150 份脱敏历史样本的金标准数据集。
2. 为每份样本标注：
   - 核心结构化字段是否正确
   - Top 10 试验中哪些可入组
   - 明确排除项与缺失证据项
3. 固化研究级结构化字段清单，至少覆盖：
   - 诊断
   - 病理类型
   - 分期
   - 基因突变 / PD-L1
   - ECOG
   - 年龄 / 性别
   - 既往治疗线数
   - 脑转移
   - HBV/HCV/感染
   - 肝肾功能 / 血常规
   - 妊娠哺乳
4. 为每条试验补充统一的结构化入排标签，不再只依赖长文本。

判定指标：

- 核心字段结构化准确率 >= 85%
- 病种、分期、基因三大字段联合准确率 >= 90%
- Top 10 候选召回率 >= 70%

## Phase 2：补齐人工协同闭环（第 3-4 周）

目标：让报名信息真正进入研究运营流程，而不是停在前端提交。

交付物：

1. 定义报名状态流：
   - `new`
   - `assigned`
   - `contacted`
   - `pre_screening`
   - `eligible`
   - `ineligible`
   - `closed`
2. 后端新增或稳定以下能力：
   - 创建报名记录，并绑定 `recordId + trialId + user`
   - 查询报名状态历史
   - 指派研究协调员或人工客服
   - 记录联系结果和失访原因
3. 管理端最少支持三件事：
   - 查看新报名
   - 处理缺失资料
   - 回填最终预筛结论
4. 前端新增一个轻量状态视图，让患者知道“已提交、已联系、待补资料、已完成初筛”。

判定指标：

- 报名后 24 小时内首次联系率 >= 90%
- 报名数据落库成功率 >= 99%
- 人工可在 3 分钟内完成一条预筛处理

## Phase 3：验证匹配质量并优化排序（第 5-6 周）

目标：从“能展示”提升到“值得信任”。

交付物：

1. 建立人工复核反馈回流机制：
   - 命中正确
   - 误命中
   - 证据不足
   - 排除漏识别
2. 用复核结果校正匹配权重，重点优化：
   - 既往治疗线
   - 基因条件
   - 活动性感染 / 脑转移
   - 招募状态
   - 研究中心可达性
3. 为每个候选试验生成更短、更清楚的解释语句，避免过长医学文本压垮用户理解。
4. 形成每周一次的误差复盘表，按病种拆解。

判定指标：

- Top 3 推荐命中率 >= 50%
- 明确排除项漏识别率 <= 10%
- 研究团队主观可用性评分 >= 4/5

## Phase 4：开展小范围真实试点（第 7-8 周）

目标：验证产品在真实运营里是否成立。

交付物：

1. 选择 1-2 家研究中心或 SMO 团队进入试点。
2. 先做 2 周 shadow mode：
   - 系统给结果
   - 人工照常筛
   - 对比两者差异
3. 再做 2-4 周 live pilot：
   - 系统结果作为研究协调员的初筛入口
   - 跟踪联系、补资料、转化、失访
4. 周会固定看 5 个指标：
   - 上传成功率
   - 结构化成功率
   - 匹配可用率
   - 报名到联系转化率
   - 有效预筛通过率

## 5. 前端下一步实现顺序

患者端下一阶段不应该继续堆功能，而要围绕信任和闭环补最关键的四项：

1. 报名后状态页
   - 告诉患者是否已收到
   - 是否有专人跟进
   - 当前还缺什么资料
2. 缺失证据补件清单
   - 将“缺失证据”转成用户可操作的资料清单
   - 例如“补最近一次基因检测”“补肝功能”“补出院小结”
3. 结构化病历的人工纠错入口
   - 让患者或家属能纠正明显抽取错误
4. 订阅消息或联系预期提示
   - 报名成功后给出下一步时间预期，降低不确定感

## 6. 后端必须配合的事项

下一步研究落地是否成功，关键不在前端，而在后端是否补齐以下能力：

1. 稳定的报名接口
   - 不能再依赖本地兜底保存
   - 需要正式落库、可查、可回写状态
2. 报名状态流接口
   - 患者端和管理端都要能读取
3. 人工复核记录接口
   - 记录最终入组判断、原因、证据缺失
4. 匹配反馈接口
   - 将“推荐是否正确”回流给匹配引擎
5. 指标埋点与导出
   - 至少能统计上传、结构化、匹配、报名、联系、转化

建议新增的核心数据对象：

- `application_status_events`
- `review_labels`
- `match_feedback`
- `missing_document_requests`

## 7. 推荐 KPI

建议下一阶段只盯这 8 个核心指标：

1. 上传成功率 >= 95%
2. 结构化完成率 >= 90%
3. 核心字段准确率 >= 85%
4. Top 10 召回率 >= 70%
5. Top 3 命中率 >= 50%
6. 报名后 24 小时联系率 >= 90%
7. 有效预筛通过率 >= 30%
8. 患者资料补齐完成率 >= 60%

## 8. 风险与默认策略

1. 默认不扩癌种
   - 如果在试点阶段同时扩多个癌种，误差来源会被放大，无法定位问题。
2. 默认保留人工复核
   - Treatbot 当前适合做初筛，不适合直接替代研究团队判断。
3. 默认把“排除风险”权重放在“命中推荐”之前
   - 医疗场景里，错推比漏推风险更高。
4. 默认优先建设后台运营闭环
   - 没有后台和状态流，前端再好看也只是演示层。

## 9. 建议的立即执行项

如果明天开始排期，建议按下面顺序做：

1. 后端补正式报名接口和状态流
2. 管理端补人工复核与联系记录
3. 拉 100-150 份脱敏样本做金标准集
4. 前端补报名状态页和缺件清单
5. 跑 2 周 shadow mode，对齐算法与人工结果

这五项做完，Treatbot 就会从“有前端、有 AI、有演示”进入“可试点、可复盘、可迭代”的阶段。
