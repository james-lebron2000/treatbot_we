## 0. 摘要

Q2 后端把**安全合规**（JWT、COS SSE、Audit、SSRF、状态机）、**匹配引擎核心**（A1-A9 + 结构化入排 100% 覆盖）、**OCR/队列可观测性**（DLQ、Prometheus、Sentry）三大块全部落地，线上稳定。

但仍有 **3 类阻塞**未闭环，直接卡在"从能跑到能赚"中间：

| 类别 | 表现 | 商业后果 |
|---|---|---|
| **数据出口未达 MVP** | CRO 仅有基础 CSV 导出（单 trial、字段少）；申请状态流转 API 缺失 | 无法支撑 CPA 计费 → Q3 无收入 |
| **历史密钥仍在 Git** | CI 阻挡了新泄露，但旧 commit 里的 key 未清、未轮换 | 无法过医院 / 药企法务审计 → 阻商务签约 |
| **数据治理薄弱** | model↔migration 漂移、Node 18 EOL、试验数据无定时抓取、SLO 文档空白 | DR 演练失败、试验过期信息误导用户 → 信任崩盘 |

本 PRD 把这三类拆成 **P0（2 周）+ P1（4 周）+ P2（季尾）** 三档，每条都从产品 / 用户 / 客户三个视角说清楚为什么必须做。

## 1. P0 — 必须 2 周内闭环

### P0-1 · CRO 结构化线索导出 v1（H4 商业化核心）

**修改项**
- [server/controllers/cro.js:238 `exportCroApplications`](../server/controllers/cro.js)：从单 trial CSV 升级为多 trial + JSON/CSV 双格式 + 字段集扩展。
- 新增 `cro_export_log` 表 + migration `20260504_cro_export_log.sql`。
- 新增 `assertTrialOwnership(croId, trialIds[])` 工具方法 + 单测。
- 加 `trial_applications(trial_id, status, created_at)` 复合索引。
- 新增字段集：申请ID、状态、申请时间、患者昵称、手机号（默认掩码）、年龄、性别、城市、诊断、分期、ECOG、treatment_line、基因突变、PD-L1、匹配分、匹配理由 top3、record_ids。

**目的**
线上现有 `/api/cro/exports/applications` 是 Q2 早期 demo 产物——只能导单试验、缺关键临床字段、无导出日志、字段权限粗糙。CRO 客户拿回去之后还要二次手工补充诊断信息，毫无效率提升，更不能作为 CPA 计费凭证。

**产品视角（PM）**
- 这是**商业化模式 I1（CPA 按线索收费）的物理前提**——没有结构化导出，CRO 没法验证"线索质量符合付费标准"，整个收费链路无从谈起。
- 此外，导出动作本身要审计留痕，否则未来一旦 CRO 投诉"我没拿到那批数据"或"我们没有 unmask 那批手机号"，平台无法自证。
- 字段集要兼容 CRO 现有 EDC（电子数据采集）系统的最小输入字段，降低 CRO 接入成本。

**用户视角（患者 / 家属）**
- 用户感知不到这个接口本身，但能感知**间接结果**：CRO 拿到结构化数据后能更快回电（从平均 3-5 天压缩到 1-2 天），减少"报名了没人理"的差评场景。
- 默认手机号掩码 + unmask 二次校验，意味着用户隐私在 CRO 内部流转时也有控制权——这是知情同意书里承诺的兜底。

**CRO / 客户视角**
- 拿到的不再是"申请ID + 患者昵称"这种残缺数据，而是可以**直接进入临床筛选漏斗**的结构化字段（含基因、PD-L1、ECOG、treatment_line）。
- JSON 格式可以直接对接 CRO 内部 CRM；CSV 给运营人员看。两种格式覆盖客户两类工作流。
- `cro_export_log` 让 CRO 也能反查"我们什么时候导了哪些数据"，自查合规也方便。

**完成后获益（量化指标）**
- ✅ 至少 **1 家 CRO 在 v23.0 后用 JSON 接口直接对接 CRM**（vs 现状 0 家）。
- ✅ 患者从"提交申请"到"CRO 首次联系"P50 时间 **5d → 2d**（依赖 CRO 配合）。
- ✅ 平台获得 **第一笔 CPA 收入凭证**——按月对账时 `cro_export_log + application_status_event` 可交叉验证。
- ✅ 100 条申请导出 P95 < 1.5s（性能验收）。

### P0-2 · 申请状态机 + 状态流转 API（C8 链路打通）

**修改项**
- 新增 `server/services/applicationStateMachine.js`：定义合法迁移 + `transition(appId, to, { actor, reason })`。
- 新增 `application_status_event` 表 + migration `20260505_application_status_event.sql`。
- 新增 `POST /api/cro/applications/bulk-status`、`GET /api/applications/:id/timeline`。
- 替换 `controllers/application.js`、`controllers/cro.js`、`controllers/admin.js` 中所有直接 `UPDATE status` 的代码路径，统一走 `transition()`。
- 新增 `server/services/notify.js`（先打日志桩，留接口给 P2-5 微信模板消息）。

**目的**
当前 `trial_applications.status` 字段已经有，但**只有字段没有协议**：admin 单条 PUT 直接改 status，没有合法迁移校验，没有审计事件，没有通知 hook。CRO 想批量把 50 个申请从 `contacted` 推到 `screened` 只能一条一条点击。患者侧完全看不到自己的状态历史——"申请之后就石沉大海"是当前最大的差评源。

**产品视角（PM）**
- 状态机是**整个商业化链路的数据中枢**：CPA 计费要数 `screened` 事件、转化漏斗要看每一段流失率、客户侧需要 SLA 时长统计——全部依赖 `application_status_event` 这张事件表。
- 合法迁移校验避免 CRO 误操作（例如把 `rejected` 改回 `pending` 然后重复计费）。
- bulk API 是 CRO 工作流的关键效率提升点——单试验一周 100+ 申请很常见。
- timeline API 是患者侧"申请进度"页面的后端依赖，是 F3 通知体系的前置。

**用户视角（患者 / 家属）**
- 第一次能在 H5 / 小程序里看到**自己的申请走到哪一步了**：提交 → 已联系 → 已筛选 → 已入组（或 已拒绝 / 已撤回）。每一步都有时间戳和原因。
- 解决了用户最焦虑的"我提交了，到底有没有人看？"——这是当前 NPS 主要拉低项。
- 后续接通微信模板消息后，状态变化能直接推到手机，不用主动来 App 查。

**CRO / 客户视角**
- 一次批量推 100 个状态，从原来的 30 分钟点击 → 1 个 API 调用。
- 状态流转有迁移校验，避免操作员误把入组的患者改回 pending 导致的数据混乱。
- 每个状态变更都有 `actor + reason` 留痕，CRO 内部审计 / 申办方季度审查直接出报告。

**完成后获益（量化指标）**
- ✅ CRO 平均状态流转操作时间 **30min → < 1min**（批量 100 条）。
- ✅ 患者"申请进度"页可视化覆盖率 **0% → 100%**。
- ✅ 非法状态迁移尝试 **被拦截率 100%**（防止数据脏化）。
- ✅ 为 P1-4 CPA 计费提供唯一可信数据源——`application_status_event` 行数 = 计费金额依据。
- ✅ 客户侧"沉默率"（提交后 7d 内无任何状态变更）从未知 → 可观测可改进。

### P0-3 · Git 历史密钥清理 + 全量轮换

**修改项**
- 用 `git-filter-repo` 在 mirror clone 上清理涉密路径与字符串。
- force-push main + 全员 reset + 离线 tarball 备份。
- 全量轮换 5 类 key：Kimi、ARK（Doubao）、Tencent COS、Tencent SMS、Sentry DSN。
- 新增 `docs/key-rotation-log.md` 记录每次轮换：provider / rotated_at / actor / new_key_fingerprint / old_key_revoked_at。
- GitHub Actions Secrets 全部更新；本地 `.env.production` 删除（按 `docs/secrets-playbook.md` 流程）。

**目的**
Q2 已经把 CI `secret-scan` 闸门装好（commit `e8bd042`），新的提交泄露不了。但是**老仓库的历史 commit 里 key 还在**——任何人 `git clone` 之后 `git log -p | grep sk-` 就能拿到旧 Kimi key。Kimi key 没轮换 = 现在还能调用我们的账号烧钱。这是悬而未决的"已泄露"事件，不是潜在风险。

**产品视角（PM）**
- 没清干净 = 任何医院 / 药企法务尽职调查直接 fail。商务团队近期跟某 CRO 谈合作，对方法务部门要的就是"过去 12 个月密钥泄露事件清单 + 处置证据"。
- 这是**法务-商务-技术三方都需要"已闭环"证据**的合规事项。`key-rotation-log.md` 是商务对外可出示的证据。
- 同时也是 ICP 备案 / 等保测评（I4 商业合规）的前置——没有密钥管理流程过不了等保二级。

**用户视角**
- 用户感知不到，但**间接保护**：泄露的 Kimi key 一旦被滥用、被 Kimi 平台封号，会导致全平台 OCR 失效——所有人病历无法解析。轮换 + 历史清理避免"被别人用我们 key 干坏事 → 平台被服务商连坐"。

**CRO / 客户视角**
- 法务部门要求平台提供"密钥管理流程文档 + 最近一次轮换证据"，是商务签约前的必查项。
- 给 CRO 的合作合同里"信息安全条款"现在可以加一条"key 每 90 天轮换一次"——是合规升级，不是单纯技术债。

**完成后获益（量化指标）**
- ✅ `git log --all -p | grep -E "sk-[a-zA-Z0-9]{20,}|AKID[0-9A-Z]{16}"` **0 命中**（vs 当前 ≥10 命中）。
- ✅ 5 个 provider × 1 次以上轮换全部入账。
- ✅ 商务团队拥有可向客户出示的"密钥事件已闭环"证据文件。
- ✅ 后续每 90 天可按相同流程例行轮换，把这件事变成 SOP。

### P0-4 · models / migrations 漂移收口

**修改项**
- `server/scripts/migrations/20260506_baseline_drift_fix.sql`：补齐 `geneRequired_cache`、`status_live` 等遗漏字段。
- `server/app.js` 启动期校验：生产必须 `SEQUELIZE_SYNC_DISABLED=true`，否则启动失败。
- `.github/workflows/deploy.yml` 加 `migration-check` job：clean MySQL → migrate → test → 失败阻 deploy。
- GitHub Actions Secrets 加 `SEQUELIZE_SYNC_DISABLED=true`。

**目的**
最近几次新增字段只动 model 没补 migration。生产数据库目前能跑是因为 `sequelize.sync({ alter:true })` 在帮忙——但这个机制**不能在生产用**：它会偷偷改表结构、可能丢数据、无法审计。一旦哪天关掉 sync 或换台机器，整个数据库结构对不上代码，崩盘。

**产品视角（PM）**
- 这是**灾难恢复（DR）能力**的隐性缺口——出事时备份还原后启动不了 = 数据回得来服务起不来 = 实际不可恢复。
- 也是新员工 onboarding 障碍：新人本地起服务，model 和 DB 对不上，第一天就卡住。
- 持续部署的"快速回滚"前提是任意 commit 的 schema 都明确——现在做不到。

**用户视角**
- 用户感知不到。但万一 DR 时还原失败，用户面临的是"App 完全打不开"——这是产品最差结果。

**CRO / 客户视角**
- CRO 关心的是 SLA：可用性 99.5% 承诺背后必须有 DR 能力。`migration-check` CI 是 SLA 兑现的技术基础。
- 客户法务可能会要求平台出示"灾难恢复预案 + 演练记录"，这条是预案前置。

**完成后获益（量化指标）**
- ✅ 干净 MySQL 容器 → `npm run db:migrate` → `npm test` 全绿。
- ✅ `git grep -nE "sync\(\{.*alter"` 仅在 dev / test 路径出现。
- ✅ 新员工 onboarding 启动时间 **从 4h（含手动建表）→ 30min**。
- ✅ DR 演练首次可走通（具体演练在 P2-3）。
- ✅ CI 多一道闸门，未来任何 model 变更必须配 migration 才能合 main。

## 2. P1 — 4 周内交付

### P1-1 · 试验数据每日抓取（G1 + Q2 P0-4 收尾）

**修改项**
- 新增 `server/jobs/trialCrawler.js` + `server/services/clinicalTrialsClient.js`（封装 ClinicalTrials.gov v2 API）。
- 新增 `trial_change_log` 表 + migration `20260518_trial_change_log.sql`。
- 新增 `trial_crawl_failures` DLQ 表（复用 OCR DLQ 模式）。
- 新增 `GET /api/admin/trials/health`：列 ≥14d 未刷新 + 最近 30 条变更。
- `server/app.js` 注册 cron `0 3 * * *`（北京时间）。

**目的**
当前 `trialFreshnessJob` 能把陈旧试验降权——但**没有任何作业去拉新数据**。`last_verified_at` 永远停在"导入日"。后果：已停招的试验仍出现在匹配结果里，用户报名→ CRO 那边说"早关了"，体验崩盘 + 信任崩盘。

**产品视角（PM）**
- 试验数据是 Treatbot 的**第一性资产**——数据陈旧 = 平台失效。Q2 的 freshness 标记只是降权，没解决"数据从哪来"。
- 上游有公开 API（ClinicalTrials.gov v2）+ 中国临床试验注册中心，必须接通其中一个作为 source of truth。
- Admin "试验健康度"页是运营人员的日常工具——每天扫一眼哪些试验长时间没刷新、哪些 status 变了、要不要下架。

**用户视角**
- "我报名的这个试验早就关了！" 是当前用户负面反馈 top 2。每周至少 1-2 起。
- 接通后用户搜到的、匹配到的、报名的试验都是**最近 7 天验证过仍在招募**的——直接消除"白报名"场景。

**CRO / 客户视角**
- CRO 自己的试验状态变更（暂停 / 关闭）能在 24h 内同步到平台，减少"我们关了平台还在导线索"的对账纠纷。
- 平台自动跟踪 ClinicalTrials.gov 的更新，CRO 不用每周手工通知"这条改了那条加了"。

**完成后获益（量化指标）**
- ✅ 试验数据 `last_verified_at` 中位数 **从未知（旧）→ ≤ 7 天**。
- ✅ "报名后才发现已停招"投诉率 **每周 1-2 起 → 0**。
- ✅ Admin 试验健康度页可视化 + 一键下架，运营效率显著提升。
- ✅ 至少 1 条 closed 试验从 `recruiting` 自动 → `deprecated`（验证链路通）。

**用户视角**

- 减少"你这病没法匹配"的挫败体验。

**CRO / 客户视角**
- 泛瘤种 / 篮子试验（basket trial）的 CRO 现在能拿到更多潜在线索，平台对这类客户的价值显著上升。
- 国际多中心试验（用英文描述）首次能被中文患者发现。

**完成后获益**
- ✅ 5 条已知泛瘤种试验 × 3 类患者矩阵 **15/15 全命中**。
- ✅ 平均匹配试验数（无基因患者）**+30~50%**。
- ✅ 新词条由产品 / 临床顾问直接维护文档，工程不参与（流程效率）。

### P1-3 · 多病历 active 切换 API（PRD-Q2 P1-5 后端段）

**修改项**
- migration `20260520_medical_records_active.sql`：`medical_records.is_active TINYINT(1)`，回填每个 user 最新一条为 1。
- 新增 `PUT /api/medical/records/:id/activate`（事务内确保同 user 唯一）。
- `controllers/match.js` 默认走 `is_active=1`；保留 `?recordId=xxx` 显式覆盖。
- 新增 `tests/medicalActiveSwitch.test.js`。

**目的**
当前 `patientStore` 只保留 1 条结构化记录——用户上传第二份病历（例如新一次的 CT 报告），第一份会被覆盖。后果：用户不敢上传新病历，怕丢数据 → 平台无法跟踪病情进展。

**产品视角（PM）**
- 多病历是**长期用户留存**的基础设施——Treatbot 不是一次性工具，应该跟着用户的治疗历程持续更新。
- active 机制让用户对"当前用哪份做匹配"有控制权，避免"上传新报告但匹配结果没变"的困惑。
- 为后续"病历时间线"产品功能（已有 timelineService 雏形）打底。

**用户视角**
- 用户能放心地一次次上传新报告，平台能看到病情进展（基线 → 治疗中 → 治疗后）。
- "切换 active" 让用户能用旧病历回溯"如果当时我的状态不变，能匹配到什么"——临床决策辅助。

**CRO / 客户视角**
- CRO 拿到的患者档案是**完整时间序列**而不是单点快照——筛选时能判断"这个患者最近一次评估 ECOG 是 1，比 6 个月前的 2 改善了"。
- 入组评估的临床价值显著提升。

**完成后获益**
- ✅ 用户平均上传病历数 **1.0 → ≥ 2.0**（衡量持续使用）。
- ✅ 全局 active 唯一性测试通过（并发 20 次切换）。
- ✅ 为 timeline 产品功能解锁后端依赖。

### P1-4 · CPA 计费埋点 + 月度对账（I1 第一刀）

**修改项**
- migration `20260522_cpa_pricing.sql`：`cro_companies` 加 `cpa_price DECIMAL(10,2)` + `cpa_qualified_status enum('screened','enrolled')`。
- T0-2 状态机 `transition()` 中 emit `prom-client` counter `cro_qualified_lead_total{cro_id, trial_id}`。
- 新增 `server/services/billing.js` + `GET /api/admin/billing/summary?month=YYYY-MM&format=csv|json`。
- 新增 `tests/billing.test.js`。

**目的**
状态机（P0-2）打通后，**真正把"合格线索"变成钱**的是这一步。否则即使 CRO 开始付费，平台对账靠人工 SQL，月底容易出差错。

**产品视角（PM）**
- 这是**收入闭环的最后一公里**——商业模式 I1 跑通的标志就是"按月给 CRO 出账单 → 收款"。
- 不同 CRO 的单价不一样（取决于试验难度、患者稀缺度），用 `cro_companies.cpa_price` 灵活定义。
- "qualified_status" 可配置（screened 还是 enrolled 才计费）支持不同合作模式。

**用户视角**
- 用户感知不到。但商业化跑通后平台才有钱投入产品改进——长期看与用户利益正相关。

**CRO / 客户视角**
- 月底拿到的对账单是**逐条可追溯**的："这条线索的 status_event 在哪天、谁操作、从哪个 status → screened"。
- 杜绝平台多算 / 客户少认的扯皮。
- 客户还能要求 CSV 拿去内部财务审计，3 分钟出账单的工作流。

**完成后获益**
- ✅ **第一笔可正式开票的 CPA 收入产生**（依赖商务）。
- ✅ 对账时 SQL 手算 vs API summary 数字 100% 一致。
- ✅ 客户对账时间 **数小时 → < 5 分钟**。

### P1-5 · Node 18 → Node 20 LTS 升级

**修改项**
- `server/Dockerfile`：`node:18-bookworm-slim` → `node:20-bookworm-slim`（两处 stage）。
- `.nvmrc` 同步。
- `package.json` 加 `"engines": { "node": ">=20.0.0" }`。
- staging 跑 48h 灰度后切生产；保留 rollback 镜像。

**目的**
Node 18 EOL 是 2025-04（已过去一年）。继续用 = 没有安全补丁、没有性能改进、未来某天 npm 包陆续 drop 支持。

**产品视角（PM）**
- 这是**技术债清理**——不做出不了事，做了也不增加用户可见价值，但拖久了会变成"想升级时一堆依赖不兼容"的大坑。
- 顺便享受 Node 20 的性能提升（V8 11，async hooks 更快），对 P95 有几个 ms 的免费优化。

**用户视角**
- 几乎无感。可能 OCR 队列处理略快几个百分点。

**CRO / 客户视角**
- 客户法务可能在年度安全评估里看 runtime 版本——支持中的 LTS 是基本要求，EOL 版本会被打勾"不合规"。

**完成后获益**
- ✅ `node -v` 在容器里 `v20.x`。
- ✅ 67+ 测试全绿，无回归。
- ✅ 解锁未来一些 Node 20+ only 的依赖（例如新的 pdf 解析库）。
- ✅ 法务安全评估"runtime 版本"项过关。

### P1-6 · Admin RBAC 三角色（PRD-Q2 P2-2 提前）

**修改项**
- migration `20260525_admin_role.sql`：`admin_users.role ENUM('super','ops','cro_liaison')`。
- `middleware/adminAuth.js` 加 `requireRole(['super','ops'])`。
- 敏感接口加角色门：PII unmask（T0-1）、billing summary（T1-4）只允许 super。
- `admin_audit_log` 写入加 role 字段。
- 新增 `tests/adminRbac.test.js`。

**目的**
当前 admin token 单一角色——任何 admin 都能看 PII + 导出。运营 / 商务 / 工程混用一个角色，意味着实习生能看用户身份证号。

**产品视角（PM）**
- 这是**最小权限原则**的基础设施——和 P0-3 密钥清理一起构成"内部访问治理"完整画面。
- 对内合规、对外法务尽调都需要这条。
- 三角色定义对齐组织实际：super = 工程 leader、ops = 运营、cro_liaison = 客户关系。

**用户视角**
- 用户感知不到，但**间接保护**：运营人员不再能批量看身份证号 → 减少内部数据泄露面。

**CRO / 客户视角**
- 客户法务关心的"谁能看到我们的患者数据" 现在有清晰矩阵图可出示。
- 比"所有 admin 都能看"的现状显著加强。

**完成后获益**
- ✅ 三角色 × 4 关键接口矩阵单测全绿。
- ✅ 至少创建 1 个 ops + 1 个 cro_liaison 账号上线验证。
- ✅ 内部数据泄露面收窄（可量化为：能看 PII 的账号数 N → 1-2 个 super）。

- **获益**：报名→撤回率显著下降（待测）。

### P2-7 · 试验收藏 API

- **修改项**：`POST/DELETE/GET /api/trials/:id/favorite`。
- **目的**：让用户能把感兴趣但暂未报名的试验存起来。

- **CRO 视角**：收藏数是 trial 受欢迎度的早期信号。
- **获益**：用户回访率提升；匹配排序更个性化。
