## 2. P0 — 必须在下个迭代（2 周内）修复

### P0-1 · OPENAI_API_KEY 等密钥在版本库 / 镜像中暴露
- **现象**：`.env` / `.env.production` 被检索命中包含明文 key；docker build 未使用 BuildKit secrets。
- **影响**：密钥泄露 → 被第三方调用计费；合规风险。
- **方案**：
  1. 立即从 Git 历史清理（git-filter-repo），重置所有外部 key（Kimi、OpenAI、COS SecretKey、SMS）。
  2. 生产：改用 Docker secret / 部署机 systemd EnvironmentFile。
  3. CI：加 pre-commit `detect-secrets`。
- **验收**：`git log -p | grep -i 'sk-'` 无结果；`docker inspect treatbot-api` 不含密钥环境变量明文。
- **工作量**：S（1d）。

### P0-2 · 腾讯云 COS 上传病历对象未启用 SSE-加密 + 默认 ACL 过宽
- **现象**：`server/services/cos.js` 上传时未显式 `x-cos-server-side-encryption`；存储桶默认 ACL `private`，但预签名 URL TTL 未限制导致链接可被复用。
- **影响**：医疗档案泄露风险（含姓名、身份证号碎片、病历影像）。
- **方案**：
  1. PutObject 加 `ServerSideEncryption: 'AES256'`；存量对象批量 rewrite。
  2. 下载改用 `getObjectUrl({ Expires: 300 })` 短期签名；后端网关转发，不暴露原 COS 域名给前端。
  3. 桶策略：限制只允许业务 role 访问。
- **验收**：COS 控制台检查任一对象含 SSE-C；线上 F12 Network 看不到 *.cos.ap-*.myqcloud.com 直连。
- **工作量**：M（2d）。

### P0-3 · Admin 导出 / 查询接口未做操作审计
- **现象**：`server/routes/admin.js` 的 `GET /api/admin/applications`、`GET /api/admin/users` 返回病历明文、姓名、手机号但无访问日志。
- **影响**：内部人员越权查询无法追溯；违反《个人信息保护法》第 51 条。
- **方案**：
  1. 新增表 `admin_audit_log(id, admin_id, action, target_type, target_id, ip, ua, created_at)`。
  2. 在 admin middleware 出口中间件层统一写入。
  3. 导出接口强制二次 MFA（可先用邮箱验证码临时过渡）。
  4. 返回 payload 对手机号 / 身份证做掩码（尾号 4 位），仅留一个"显示完整"的按需接口。
- **验收**：抽查任一 admin GET → DB 有记录；导出含身份证需触发 MFA。
- **工作量**：M（3d）。

### P0-4 · 试验数据新鲜度未强制 — 已关闭试验仍出现在匹配结果
- **现象**：`trials.status` 无过期更新任务；crawler 只 upsert 不标注 "closed"；`matchEngine` 也未按 `status` 过滤。
- **影响**：用户报名到已停止招募的试验，投诉风险 + CRO 信任损伤。
- **方案**：
  1. Crawler 每次抓取回写 `status` + `last_verified_at`；未被抓到的试验 30d 后自动降权（score-20）、60d 标 `deprecated`、匹配 SQL 过滤 `status IN ('recruiting','active')`。
  2. Admin 界面新增"试验健康度"视图，列出 > 14d 未刷新的试验。
- **验收**：Admin 随机抽 20 条，点进去看 ClinicalTrials.gov 的 status 与我们一致。
- **工作量**：M（3d）。

### P0-5 · 报名状态机竞态 + 重复报名
- **现象**：`server/controllers/applications.js` 的 submit / cancel 未用事务 + `SELECT … FOR UPDATE`；幂等键作用域仅到 `trialId+userId`，但前端 `buildIdempotencyKey` 还加了 `recordId`，两端不一致时去重失效。
- **影响**：并发点击导致双份报名；CRO 后台噪音。
- **方案**：
  1. submit 路径包在 `sequelize.transaction`；第一步 `findOne({ where:{trialId,userId,status NOT IN (canceled)}, lock:true })`。
  2. 统一幂等键口径：前后端都用 `user_{userId}_trial_{trialId}`，`recordIds` 走 body 而非 key。
  3. 在 `applications` 表加唯一索引 `(user_id, trial_id, status_live)` 的虚拟列或部分索引。
- **验收**：Apache Bench 并发 20 条同一 trial+user 的 POST，DB 里只有 1 条 `pending`。
- **工作量**：M（2d）。

### P0-6 · `fetchImageAsDataUrl` SSRF 风险
- **现象**：`server/services/ocr.js` 里 Kimi fallback 从 COS URL 下载图像转 base64，未校验主机白名单。
- **影响**：攻击者可构造 COS 预签名 URL 指向内网（unlikely 但存在链）。
- **方案**：允许域名列表 `['cos.ap-*.myqcloud.com']`；强制 http(s) + 超时 5s + 响应大小上限 15MB。
- **验收**：单测：传入 `http://169.254.169.254/latest/meta-data` 抛错。
- **工作量**：S（0.5d）。

### P0-7 · pdf.js / 图片裁剪库 首屏大包 549KB 未懒加载
- **现象**：`web/dist/assets/pdf.*.js` 被 UploadView 同步 import。
- **影响**：Treatbot Web 首屏多下 ~500KB gzip，弱网下跳出率升高；用户首页（MatchesView）不需要 pdf.js。
- **方案**：改为 `const { getDocument } = await import('pdfjs-dist/webpack')`；wrap 在预览 PDF 的方法里。
- **验收**：`vite build` 后 initial chunk < 250KB gzip；Network 确认 pdf chunk lazy。
- **工作量**：S（0.5d）。

## 3. P1 — 4 周内闭环（体验与扩展性）

### P1-1 · 泛瘤种 / 不限突变双向匹配仍有漏网
- 位置：`server/services/matchEngine.js:scoreRecordAgainstTrial`
- 问题：`GENE_AGNOSTIC_HINTS` 已扩展，但对"advanced solid tumor"英文表述、"晚期实体瘤"繁体、以及 `structured_inclusion.disease_tags` 数组的 any-match 判定仍不完整。
- 方案：把 `hasGenericCancerSignal` 拆成多源聚合（text + structured + disease_tags），增加语料字典到 `docs/trial-vocabulary.md`；加 10 条 snapshot test。
- 验收：5 条已知泛瘤种试验对"肺癌无基因"患者全部 hit。

### P1-2 · Bull OCR 队列无死信与重试
- 位置：`server/workers/ocrWorker.js`、`server/services/ocrQueue.js`
- 问题：`attempts` 只配了 3 次硬重试；Kimi 429 / 超时未退避；失败 job 无 DLQ，Redis 溢出时会被 eviction 静默丢失。
- 方案：
  1. `defaultJobOptions: { attempts: 5, backoff: { type: 'exponential', delay: 5000 } }`
  2. `events: 'failed'` 写到 `ocr_job_failures` 表，Admin 页有重试按钮。
  3. Prometheus metric `ocr_queue_waiting / failed`。
- 验收：强制拔 Kimi key，jobs 平滑进入 failed 表，不丢请求。

### P1-3 · 幂等 / 响应信封不统一
- 现象：`/api/applications` 返回 `{code, message, data}` 而 `/api/demo/*` 直接裸对象；前端 `unwrap` 有多层兜底。
- 方案：落一个 `server/middleware/responseEnvelope.js`，所有新接口必须 `res.ok(data)`；前端 `api.ts` 简化掉 `ApiResponse & T` 的脏泛型。
- 验收：新增接口 lint 强制；旧接口灰度迁移，至少 demo / match / application 三条主链一致。

### P1-4 · 无 token 刷新 + 401 硬跳转
- 位置：`web/src/services/api.ts:22-33`
- 问题：JWT 3 小时过期后用户填到一半表单会被踢回 /login，填写数据丢失。
- 方案：
  1. 后端新增 `/api/auth/refresh`（短 access + 长 refresh）。
  2. 前端 interceptor 遇 401 先尝试 refresh，成功则重放原请求；两次仍失败再跳转；跳转前把 `from` 存到 sessionStorage 恢复。
- 验收：手动改 token 过期时间 1min 模拟漂移，不退出当前页。

### P1-5 · 病历"多份管理"空缺
- 现象：`patientStore` 只保 1 条 `structuredRecord`；用户上传第二份病历会覆盖第一份。
- 方案：新增 `/records` 管理页（列表 + 切换 active + 软删除）；复用 `GET /api/medical/records`（已存在）。
- 验收：上传 3 份病历，列表可切换，匹配基于 active 记录；切换后 MatchesView 自动 refetch。

### P1-6 · SMS 反刷强度不足
- 现象：`/api/auth/send-code` 仅 60s 同手机号锁；同 IP 可批量刷号码。
- 方案：per-phone 日上限 5 次、per-IP 小时上限 20 次、加滑块（腾讯防水墙）在 Treatbot Web 侧。
- 验收：curl 脚本 10 分钟对 100 个手机号发码 → 有一半被拦。

### P1-7 · Treatbot Web 与小程序文案 / 能力漂移
- 现象：`pages/upload/upload.js` 小程序版错误提示未跟进 Treatbot Web 的 rate_limit 分类；没有"手动录入"入口。
- 方案：抽一份 `shared/copy/upload.json` 两端共用；小程序加同样的手动录入跳转页。
- 验收：两端截图对比，错误与主路径一致。

### P1-8 · models 与 migrations 漂移
- 现象：最近几次加列（如 `geneRequired_cache`、`status_live`）只改了 model，未生成 migration；新环境 `sync()` 通过但 `sequelize-cli db:migrate` 不完整。
- 方案：补历史 migration；CI 跑 `db:migrate:status` 校验；禁用 `sequelize.sync({ alter })` 在生产。
- 验收：干净数据库执行 `db:migrate` 后所有测试通过。
